# Audyt projektu — 2026-05-04

> Skoncentrowany przegląd: bezpieczeństwo, UX page-buildera, kompletność funkcji, język.
> Zakres: `server/` (Laravel 12 + Inertia admin) + `client/` (Next.js 16 storefront).

---

## TL;DR

Projekt jest **bardzo duży i kompletny funkcjonalnie** — 118 kontrolerów, 109 modeli, 174 migracji, 38 typów bloków page-buildera, Shopify-like (variants, metafields, smart collections, polymorphic tags, model versioning). To plus, ale jednocześnie:

- **3 realne luki bezpieczeństwa** (jedna SQL injection od strony admin, dwie XSS przez `<style>` w page-builderze).
- **Sidebar admina pomija ~10 modułów** — wiele ficzerów istnieje tylko w routes, niedostępne z UI.
- **Page builder ma narzut funkcji enterprise** (approval workflow, versioning, custom CSS, animations, scheduled publish, export/import) — to dobre, ale **nie ma wizualnego edytora live** ("WYSIWYG") jak Shopify Sections / Gutenberg. Edycja w bocznych formularzach + osobny preview.
- **51 stron admina (~34%) bez i18n** — wszystkie napisy zahardkodowane po angielsku.
- **`laravel/ai` w composer ale w ogóle nie używany** — 4MB zależności na nic.

## 1. Bezpieczeństwo — KRYTYCZNE

### 🔴 1.1 SQL injection w `CustomReportBuilderService::applyGroupBy/applyFilters`

**Plik:** [server/app/Services/CustomReportBuilderService.php:172-180](server/app/Services/CustomReportBuilderService.php#L172)

```php
$query->addSelect(DB::raw(sprintf('DATE(%s) as %s_group', $field, $field)));
$query->groupBy(DB::raw(sprintf('DATE(%s)', $field)));
```

`$field` pochodzi z `report->group_by` — admin może wstawić tam dowolny string (`CustomReportController::store` waliduje tylko `'group_by' => ['nullable', 'array']`, nie sprawdza zawartości). Nazwa kolumny trafia do `sprintf` bez walidacji whitelisty → SQL injection (np. `1) UNION SELECT password FROM users--`).

To samo w `applyFilters` (line 132+) — pole `field` używane w `where($field, ...)` — Eloquent waliduje column names, ale nadal można obejść przez `whereBetween('created_at', ...)` jeśli ktoś podłączy raw.

**Rekomendacja:** dodać whitelistę pól z `$dataSources[$ds]['fields']` przed wywołaniem `applyFilters`/`applyGroupBy`. Validate `group_by.*.field` w FormRequest (utwórz `StoreCustomReportRequest`).

**Ryzyko:** średnie — wymaga roli admin, ale admin/editor-injection jest legitymation issue (privilege escalation z editor → admin przez delete/update na arbitrary table).

### 🔴 1.2 XSS przez `_custom_css` blok w page-builderze

**Plik:** [client/components/page-builder/section-renderer.tsx:8-14](client/components/page-builder/section-renderer.tsx#L8)

```ts
function sanitizeCss(css: string): string {
    return css
        .replace(/expression\s*\(/gi, '')
        .replace(/javascript\s*:/gi, '')
        .replace(/<\s*script/gi, '')
        .replace(/url\s*\(\s*['"]\s*javascript/gi, '');
}
```

Sanitacja regexem trywialnie obchodzona:
- `expre\ssion(` (CSS escape)
- `expression(` (Unicode)
- `JaVaScRiPt:` — wprawdzie regex jest `gi`, ale inne escapes (`\6A avascript:`) przejdą.
- `</style><script>alert(1)</script>` — regex `/<\s*script/gi` blokuje ale `</style><<script>` może obejść (zależy od HTML parsera).
- `behavior:url(...)` — IE-specific, nadal pomijane.
- `@import url('//evil.com/x.css');` — w ogóle niefiltrowane.

**Rekomendacja:** użyć dedykowanego CSS parsera (np. `csstree` z whitelistą property/value) **albo** ograniczyć `_custom_css` do classów `class:` (klasy whitelistowane), **albo** dodać CSP `style-src 'nonce-...'` i nie używać inline `<style>` w ogóle.

---

### 🔴 1.3 XSS w bloku `custom_html` — niesanityzowany `<style>` tag

**Plik:** [client/components/page-builder/blocks/custom-html.tsx:10](client/components/page-builder/blocks/custom-html.tsx#L10)

```tsx
{cfg.css && <style dangerouslySetInnerHTML={{ __html: cfg.css }} />}
```

`cfg.css` jest renderowany **w ogóle bez sanitacji**. Editor może wstrzyknąć `</style><script>...</script>` lub `expression(...)` lub `behavior:url(...)`.

**Rekomendacja:** sanityzować `cfg.css` przez ten sam (poprawiony) `sanitizeCss`, **i dodatkowo** escape `</style>` w treści. Najlepiej zastąpić cały `custom_html` block **gotowymi widgetami** (Embed, Video, HTML iframe sandbox).

### 🟡 1.4 `UpdatePageBuilderRequest` — luźna walidacja

**Plik:** [server/app/Http/Requests/Admin/Cms/UpdatePageBuilderRequest.php:39-44](server/app/Http/Requests/Admin/Cms/UpdatePageBuilderRequest.php#L39)

```php
'sections.*.section_type' => ['required', 'string'],
'sections.*.blocks.*.type' => ['required', 'string'],
'sections.*.blocks.*.configuration' => ['sometimes', 'array'],
```

Brak walidacji że `section_type` ∈ keys(`config('cms.sections')`) i `block.type` ∈ `PageBlockTypeEnum`. Można utworzyć blok dowolnego typu (zostanie zachowany w DB) — atak DoS przez wstrzykiwanie ogromnych struktur do `configuration` (brak limitu rozmiaru).

**Rekomendacja:** dodać `Rule::in(array_keys(config('cms.sections')))` i `Rule::enum(PageBlockTypeEnum::class)`. Limit `configuration` na rozsądny rozmiar (np. 64KB JSON).

---

### 🟡 1.5 `authorize()` zwraca `true` bez sprawdzania policy

**Plik:** wszystkie FormRequests w `Admin/Cms/`

```php
public function authorize(): bool { return true; }
```

Polega się tylko na middleware `admin` + `role:editor|admin`. Nie ma policy per-page (`PagePolicy::update`) — **każdy editor może edytować każdą stronę** w systemie. Brak ownership/per-tenant separation.

**Rekomendacja:** dodać `PagePolicy` z `update`, `delete`, `publish`, `approve` — w `authorize()` użyć `$this->user()->can('update', Page::find($this->route('page')))`.

---

### 🟢 1.6 Co jest dobre

- Rate limiting: `api.strict` (10/min na IP), `api.public` (60/min), `api.auth` (300/min). Sensowne wartości.
- Sanctum + Fortify + 2FA — solidny stack auth.
- CSRF chronione w sesji web (Inertia), `force.json` middleware na API.
- Webhooki PayU/P24 mają weryfikację podpisu (MD5/SHA256).
- IdempotencyMiddleware dla checkout — jest.
- DOMPurify (sanitizeHtml) używany w wielu blokach (rich-text, two-columns, three-columns, accordion, tabs).

### 🟡 2.4 Page Builder — duplikacja logiki update

**Plik:** [server/app/Http/Controllers/Admin/Cms/PageBuilderController.php:89-141](server/app/Http/Controllers/Admin/Cms/PageBuilderController.php#L89)

`update()` ma **dwie ścieżki**:
1. Nowa (z `snapshot`) → `PageBuilderSyncService::sync()` (zachowuje IDs, smart diff)
2. Stara → `delete()` wszystkich sekcji+bloków + utworzenie od nowa (gubi IDs, relacje, media!)

Stara ścieżka jest **niebezpieczna**: po zapisie tracimy ID bloków → URLs `?block={id}` w admin-bar przestają działać, traci się relacje (eager-load kasowanych modeli przed re-create), tracimy ActivityLog historii.

**Rekomendacja:** usunąć starą ścieżkę. Wymaga migracji frontendu na zawsze-snapshot tryb. Sprawdzić ile miejsc używa starej ścieżki — być może zero.

## 3. Page Builder vs Shopify / WordPress (Gutenberg)

### Co jest jak Shopify (sections + blocks):
✅ Sections → Blocks hierarchia  
✅ Reusable Blocks (jak Shopify Sections globalne)  
✅ Section Templates (gotowe układy)  
✅ Schema-driven config per block (`config/blocks.php`)  
✅ Settings na poziomie sekcji (variants, layouts)  
✅ Polymorphic relations (product, category, brand, page, menu, form, faq) — silne  
✅ Metafields (Shopify-like)  
✅ Smart Collections (Shopify rule-based)  
✅ Draft/Publish + Versioning + Approval Workflow  
✅ Scheduled Publishing (B2)  
✅ Export/Import JSON (B7)

### Czego brakuje (Shopify/Gutenberg ma, my nie):

| Feature | Shopify | Gutenberg | My |
|---|---|---|---|
| **Live WYSIWYG** (edycja inline na stronie) | ✅ | ✅ | ❌ — formularze w bocznym panelu |
| **Drag-and-drop bloków bezpośrednio na canvas** | ✅ (sortable w iframe) | ✅ | ❌ — sortable tylko w liście |
| **Multi-block selection** | ❌ | ✅ | ❌ |
| **Block Library Search z preview** | ✅ | ✅ | ✅ (block-type-picker.tsx — OK) |
| **Theme presets / globalne style** | ✅ | ✅ (theme.json) | 🟡 (Themes są ale nie wpływają na block rendering) |
| **Responsive controls per block** | ✅ | ✅ | ❌ (mobile preview jest, ale brak `mobile_only`/`desktop_only` w config) |
| **AI content generation (alt-text, descriptions)** | ✅ (Shopify Magic) | ❌ | ❌ — `laravel/ai` zainstalowane ale niewykorzystane |
| **A/B testing sekcji** | ✅ | ❌ | ❌ |
| **Reusable Block Library Marketplace** | ✅ | ✅ | ❌ |

### Czy edycja jest **intuicyjna i prosta**?

**Nie do końca.** Główne tarcia:

1. **Edycja w bocznych formularzach, nie inline.** Użytkownik klika blok → otwiera się formularz po prawej, musi przewijać między formularzem a podglądem (split-view łagodzi, ale nadal dwa konteksty).

2. **Brak "block toolbar"** unoszącego się nad blokiem (bold/italic/link/move/delete) jak w Gutenbergu/Notion. Jest tylko `BlockCard` z menu kontekstowym w bocznym panelu.

3. **38 typów bloków** w jednym miejscu — Block Type Picker ma kategorie i search, ale przy pierwszym użyciu paraliż wyboru.

4. **Custom CSS / Custom HTML / Animations / Lock / Approval / Scheduled Publish / Custom Classes / Custom IDs** to pozycje w panelu Advanced — **dla zwykłego edytora to za dużo**.

5. **Sekcja vs Blok** — edytor musi rozumieć różnicę. W Shopify to jasne (sekcja = strip, blok = element wewnątrz). U nas też, ale brak intro / onboardingu.

## 8. Q&A

**Czy ten CMS jest jak Shopify / WordPress (Gutenberg)?**

— **Backend: tak**, mocniej niż większość. Polymorphic blocks, sections, metafields, smart collections, versioning, approval workflow — to Shopify-tier.

— **Frontend (admin UX): nie do końca**. Brakuje:
- Live WYSIWYG w iframe (Shopify Sections theme editor)
- Inline edit z block toolbar (Gutenberg)
- Globalnych design tokens / theme.json
- AI-assist (Shopify Magic)

To są największe braki w "intuicyjności". Funkcjonalnie wszystko jest, ale UX edycji to **forma + lista**, nie **canvas + inline**.

**Czy ze wszystkiego korzystamy?**

— **Backend: 95%** używane. Wyjątki: `laravel/ai` (0%), affiliate (50% — backend bez frontendu).

— **Admin UI: ~70% widoczne**. ~10 modułów istnieje tylko w routes, niedostępne z sidebara → faktycznie nieużywane.

— **Block types: 38 zaimplementowane**. Bez analytics nie wiadomo, ale doświadczenie podpowiada że top 10 (hero, text, products, image, CTA, columns, FAQ, gallery, video, newsletter) pokrywa 90% przypadków. Reszta = długi ogon.

**Co możemy uprościć?**

— **Najszybsza wygrana**: konsolidacja sidebara (5 grup) + onboarding checklist (1 dzień pracy, 10× lepsze UX).

— **Średnia wygrana**: schowanie Advanced/Custom CSS/Animations za feature flagiem dla nie-technicznych edytorów (½ dnia).

— **Duża wygrana**: inline-edit `rich_text` na canvasie (3-5 dni, ale game-changer dla edytora).
